# 轻量双端部署平台设计文档

副标题：以 OpenClaw Deploy 为参考实现，可复用到其他 client/server 型运维与配置分发项目

## 1. 文档目标

本文件不再只是某一次需求的草稿，而是一份可复用的设计规范。

它解决两类问题：

1. 当前项目应该如何设计、演进、维护
2. 其他类似项目可以直接复用哪些架构与工程原则

适用场景：

- 有一个常驻客户端，部署在用户机器上
- 有一个集中式服务端，负责账号、设备、配置和状态汇总
- 客户端需要周期性上报状态，并从服务端拉取配置
- 需要单文件部署、跨平台、低依赖、低运维成本

---

## 2. 总体设计目标

### 2.1 核心目标

- 单文件部署：客户端和服务端都尽量以单个二进制交付
- 低依赖：运行时不依赖外部 CDN、不依赖 CGO、不依赖复杂中间件
- 可维护：前后端分离、模块按职责拆分、配置和状态清晰分层
- 可迁移：当前实现可服务 OpenClaw，设计本身可迁移到其他类似项目
- 可控安全：最少权限、明确认证边界、对高频数据与持久数据做分级处理

### 2.2 非目标

- 不追求一开始就上分布式、多节点、高可用
- 不强制引入 Gin、ORM、消息队列、Redis 等额外组件
- 不把所有实时数据都写盘
- 不把 Web UI 依赖外网资源

---

## 3. 总体架构

推荐目录结构：

```text
.
|-- client/
|   |-- main.go
|   |-- backend/
|   |-- frontend/
|   `-- build.bat
|-- server/
|   |-- main.go
|   |-- backend/
|   |-- frontend/
|   |-- swaggerui/
|   `-- build.bat
|-- internal/
|   `-- shared/
|-- README.md
|-- README_EN.md
`-- design.md
```

设计原则：

- `client` 和 `server` 是两个独立程序
- `frontend` 与 `backend` 分离，但最终一起打包进二进制
- `internal/shared` 用于放两端复用的内部工具，不对外暴露为公共库
- 文档、构建脚本、配置示例与代码一起维护，避免文档脱节

## 4. 通用设计原则

### 4.1 API 风格

- 优先使用 REST 风格 API
- 使用明确的 HTTP 方法表达语义：`GET`、`POST`、`PUT`、`DELETE`
- 路由应显式、可搜索、可测试
- “自动路由”不是目标本身，“路由可维护且不冲突”才是目标

通用建议：

- 小项目优先标准库 `net/http`
- 只有在中间件、分组、生态真的有收益时再引入 Gin 等框架
- 路由注册应集中管理，避免分散在多个隐式入口里

### 4.2 前后端分离，但最终单文件交付

- 前端资源本地化，不使用 CDN
- HTML/CSS/JS 与后端逻辑分离开发
- 构建产物通过 `embed` 打进二进制

这个模式的优点可以直接复用到其他项目：

- 部署简单
- 版本一致，不会出现前后端资源不匹配
- 离线环境、内网环境更稳定
- 不依赖前端静态文件额外发版

### 4.3 配置与状态分离

必须明确区分两类数据：

- 配置：端口、监听地址、SMTP、Token、路径、超时等
- 状态：用户、设备、会话、绑定关系、待下发配置、运行时状态

推荐做法：

- 配置保存在独立配置文件
- 状态保存在独立状态库
- 配置支持热加载
- 状态由业务逻辑和数据库统一维护

这样做的优点：

- 配置变更不会污染业务状态
- 配置热加载更容易实现
- 状态库可独立迁移、备份、压缩、修复

### 4.4 冷数据与热数据分离

这是本项目现阶段最值得复用的一条设计经验。

应把数据分成两类：

- 冷数据：用户、设备归属、备注、待下发配置、注册开关
- 热数据：最后心跳时间、在线状态、CPU/内存占用、瞬时网络状态

规则：

- 冷数据持久化
- 热数据优先放内存
- 热数据只在必要时上报并展示，不必每次写盘

这样做的优点：

- 降低磁盘写放大
- 降低 SQLite 锁竞争
- 降低 SSD 寿命消耗
- 让高频心跳从“存储问题”变成“内存问题”

这个原则对所有有“设备心跳/Agent 上报”的项目都成立。

### 4.5 默认使用无 CGO SQLite

默认数据库建议：

- `modernc.org/sqlite`

原因：

- 无 CGO，跨平台编译更简单
- 单机部署成本低
- 易备份、易迁移、易排查
- 足够覆盖中小型控制台类项目

数据库设计原则：

- 开启外键
- 主键明确，不依赖模糊字段
- 唯一约束要覆盖用户名、邮箱、设备码这类天然唯一值
- 删除用户时要同步处理设备归属、会话、重置令牌，避免孤儿数据
- 使用事务保证一致性

通用提醒：

- SQLite 适合单机单进程控制面
- 若未来是多实例服务端共享状态，不应继续共用一个 SQLite 文件

### 4.6 数据迁移必须内建

只要状态层发生存储格式变化，就必须考虑旧数据迁移。

推荐策略：

- 新版本启动时自动探测旧状态文件
- 迁移成功后转入新状态库
- 迁移逻辑要幂等
- 迁移完成后系统应能继续启动，不要求用户手工导入

这个模式同样可复用于任何“JSON -> SQLite”“SQLite -> 新表结构”的演进过程。

### 4.7 认证边界分层

一个系统里可以同时存在多种认证方式，但边界必须清晰。

建议拆成三层：

1. 服务端 Web 会话认证
2. 服务端 API Token 认证
3. 客户端本地控制台认证

设计要求：

- 本地客户端登录不应影响客户端和服务端的后台通信
- 外部程序调用服务端 API 时，不应被迫先走浏览器登录
- 管理员能力与普通用户能力必须在前后端都做限制

这个认证分层模型对所有“本地 Agent + 中央控制台 + 自动化程序接入”的系统都适用。

### 4.8 状态同步采用“客户端拉模型”

推荐客户端与服务端之间使用：

- 心跳上报
- 周期性拉取配置

不优先使用服务端主动推送，原因是：

- NAT/防火墙环境更友好
- 客户端部署更简单
- 服务端连接管理成本更低
- 出问题时更容易排查

典型流程：

1. 客户端定期上报自身状态
2. 服务端根据设备归属与待下发配置返回响应
3. 客户端发现有新配置则写入本地并重启目标服务

这个模式的优点在很多运维项目里都成立：简单、稳、便于失败重试。

### 4.9 标识符规范化

设备码、用户名、邮箱、路径等字段经常会存在“逻辑相同但格式不同”的问题。

设计要求：

- 在系统入口统一规范化
- 存储前规范化
- 查询匹配时做兼容

例如设备识别码：

- 允许旧格式存在
- 内部统一转换为紧凑格式
- 显示与检索都基于规范化值

这类“规范化再入库”的设计，适合任何有设备码、许可证码、节点码的项目。

### 4.10 UI 原则

UI 不需要华丽，但必须清晰、一致、可维护。

建议：

- 使用中性灰白为主色
- 蓝色用于主要动作
- 红色用于危险动作
- Toast 提示统一放在右上角
- 管理台采用左侧导航、右侧详情的双栏布局
- 普通用户视图与管理员视图做明显区分

不要做的事：

- 不要把普通用户看不到或不能操作的功能仍然渲染出来
- 不要把破坏性动作放成和普通动作一样的样式
- 不要堆巨大按钮和不成比例的输入框

这组规则可以直接套用在其他运维后台或客户端控制台上。

### 4.11 安全与健壮性

必须默认考虑：

- 输入校验
- JSON 解析失败处理
- 认证失败处理
- 权限不足处理
- 路由大小写与重定向行为
- Windows / Linux / macOS 文件路径和换行差异

建议默认：

- 配置文件 UTF-8
- 源码统一 LF
- 运行时对 CRLF/LF 输入具备容忍能力
- 错误响应统一结构化输出

### 4.12 文档与接口说明同步维护

如果系统提供 API，就应同步提供：

- OpenAPI JSON
- 本地 Swagger UI
- README
- 设计文档

这样做的优点：

- 自动化调用更容易落地
- 新成员上手快
- 文档与代码的偏差更容易被发现

## 5. 当前项目的参考实现

以下内容是“通用原则”在 OpenClaw Deploy 中的具体落地方式。

### 5.1 客户端

当前客户端实现了：

- 首次启动生成本地 `config.json`
- 默认本地登录账号 `admin / admin`
- 生成并持久化设备识别码
- 自动规范化设备码
- 自动规范化 `server_url`
- 本地 Web 控制台查看与编辑 `openclaw.json`
- 周期性心跳同步
- 显示服务端配置状态、最近同步结果和连接状态
- 自动生成 `clientdeploy.sh`

客户端关键设计点：

- 本地 UI 认证和服务端心跳通信彻底解耦
- 本地配置编辑与远程配置下发复用同一份 `openclaw.json`
- 本地端口、路径、同步间隔都由 `config.json` 控制

### 5.2 服务端

当前服务端实现了：

- 注册、登录、找回密码、重置密码
- 默认管理员 `admin / admin`
- SMTP 设置
- 注册开关
- 用户管理
- 设备绑定、备注、删除、配置下发
- 管理员按用户名筛选设备
- AI Token 直接调用受保护 API
- Swagger UI 与 OpenAPI JSON
- `serverconfig.json` 热加载
- SQLite 状态持久化
- 旧版 JSON 状态自动迁移

服务端关键设计点：

- 配置文件和状态库分离
- 运行时热数据只在内存中维护
- 设备在线状态基于最近心跳和同步间隔推导
- 权限模型同时覆盖前端可见性和后端接口保护

### 5.3 当前持久化策略

当前项目将数据分为三层：

1. `serverconfig.json`
2. `server-state.sqlite`
3. 内存热状态

分工如下：

- `serverconfig.json`：端口、监听地址、AI Token、SMTP、公开 URL 等
- `server-state.sqlite`：用户、会话、设备归属、设备备注、待下发配置
- 内存热状态：在线状态、最后心跳时间、机器资源占用、当前上报配置哈希

这是本项目目前最重要的设计改进之一，也最值得在其他项目中复用。

## 6. 可复用的设计收益

本项目演进后，沉淀出了下面几条可以直接复用到其他项目的“低成本高收益模式”。

### 6.1 单文件全栈交付

适合：

- 内网控制台
- 边缘节点控制端
- 轻量运维面板

收益：

- 发布简单
- 升级简单
- 资源文件不丢失

### 6.2 配置/状态/热数据三分法

适合：

- Agent 心跳
- 设备监控
- 配置下发

收益：

- 存储成本低
- 系统一致性更好
- 性能边界更清晰

### 6.3 双认证模型

一个系统同时支持：

- 人类用户登录
- 自动化程序 Token 调用
- 本地控制台登录

收益：

- 自动化和 UI 都好用
- 权限边界清楚
- 更适合真实运维场景

### 6.4 客户端拉取配置

适合：

- 客户端在用户机器或 NAT 环境后
- 服务端不想维持大量长连接

收益：

- 网络适配性强
- 失败重试简单
- 运维成本低

### 6.5 内建迁移

适合：

- 早期快速迭代项目
- 未来会从 JSON 演进到 SQLite 或其他状态库的项目

收益：

- 降低升级门槛
- 减少人为迁移错误
- 方便后续存储重构

## 7. 后续演进建议

以下能力不是当前必须，但设计上应保留演进空间：

- 真二维码替代当前 ASCII 身份矩阵
- SQLite 增量写入，替代当前快照式持久化
- 设备热状态设置内存过期策略
- 更细粒度的审计日志
- SMTP 测试发送
- 节点批量操作
- 更严格的 API 限流与暴力破解防护
- 可选 HTTPS / 反向代理部署说明

---

## 8. 设计检查清单

以后新增功能时，至少检查以下问题：

- 这是配置、冷数据还是热数据
- 是否真的需要持久化
- 是否会产生孤儿数据
- 是否需要迁移旧数据
- 是否会影响普通用户和管理员的权限边界
- 是否会影响本地认证与远程认证的边界
- 是否仍然保持单文件部署
- 是否引入了不必要的外部依赖
- 是否能在 Windows、Linux、macOS 上保持行为一致
- README、OpenAPI、设计文档是否已同步更新

---

## 9. 当前项目结论

OpenClaw Deploy 当前已经从“功能草图”演进为一套可复用的轻量 client/server 控制面方案。

它最有价值的不是某一个 OpenClaw 功能点，而是这组可以迁移到其他项目的架构经验：

- 单文件交付
- 前后端分离但本地嵌入
- 配置与状态分离
- 冷热数据分层
- 无 CGO SQLite
- 本地认证与远程认证解耦
- 客户端拉取配置
- 内建迁移

后续无论继续服务 OpenClaw，还是迁移到其他节点管理、脚本分发、边缘设备控制、轻量运维面板项目，这些原则都可以直接复用。
